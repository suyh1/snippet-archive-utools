# 主题与操作性问题修复设计

## 背景
当前插件存在两类问题：
1. 编辑器主题切换会带动全局深浅色切换，但全局 UI 的字体、图标、浮层和控件颜色覆盖不完整，导致深色/浅色模式下可读性差。
2. 存在多个非语法层面的操作性问题，包括导入导出数据丢失、拖拽排序不完整、删除/移动后状态不同步、编辑器保存时机不稳等。

## 目标
- 统一主题应用链路，保证编辑器主题切换后插件整体明暗模式一致，且主要 UI 组件颜色完整跟随。
- 对核心操作链路做一次全面巡检，修复本地可确认的问题，优先覆盖新建、重命名、拖拽、删除、导入导出、切换片段、保存等高频路径。
- 初始化 Git 仓库，便于后续跟踪本次修复。

## 方案选择
采用保守增强方案：保留“编辑器主题驱动全局明暗模式”的现有交互，不新增独立的应用主题设置。

原因：
- 不改变用户当前心智模型。
- 能集中修复当前主题分叉问题。
- 可以把 UI 修复与操作性 bug 巡检一起收口，避免零散补丁。

## 设计

### 1. 统一主题状态来源
- 将主题明暗判断收口到 `src/utils/themes.ts`，由它提供“主题是否为深色”和“将主题应用到文档”的统一能力。
- `src/stores/settingsStore.ts` 不再维护单独的深色主题名单，避免和 `src/utils/themes.ts` 分叉。
- `src/App.vue` 与 `src/components/SnippetEditor.vue` 都通过统一主题应用函数切换全局明暗模式与编辑器主题。

### 2. 完整补齐全局样式覆盖
- 在 `src/styles/global.css` 中补齐 Ant Design Vue 相关控件在深浅模式下的颜色覆盖，包括输入框、选择器、下拉菜单、单选按钮、按钮、抽屉、弹窗、空状态、占位文字、图标、菜单项和树节点状态。
- 保持自定义组件使用统一 CSS 变量，避免局部写死颜色。

### 3. 修复数据与树操作问题
- 导入逻辑改为真正恢复文件夹层级和片段完整数据，而不是只重建壳结构。
- 增加导入时的 ID 冲突处理，并在导入后刷新选中/展开状态。
- 修复拖拽时只移动不重排的问题，确保同级文件夹和片段的 `order` 一致。
- 修复删除、复制、移动之后的顺序维护与状态清理。

### 4. 修复编辑器保存与片段切换问题
- 确保切换片段、切换 snippet、卸载组件、手动保存时都会把编辑器中的最新内容落盘。
- 复制和导出优先使用编辑器当前内容，避免复制到旧内容。
- 保持标题、描述、标签和代码的本地状态同步，减少防抖保存造成的竞态。

### 5. 验证策略
- 由于仓库当前没有现成测试基础设施，先不额外引入测试框架。
- 通过 TypeScript 类型检查和 `vite build` 做静态验证。
- 对核心链路逐项审查并结合代码路径做行为校验。

## 影响范围
- `src/utils/themes.ts`
- `src/stores/settingsStore.ts`
- `src/styles/global.css`
- `src/components/SettingsPanel.vue`
- `src/components/SnippetEditor.vue`
- `src/components/FolderTree.vue`
- `src/components/Sidebar.vue`
- `src/stores/snippetStore.ts`
- `src/stores/folderStore.ts`
- `src/utils/export.ts`
- 新增或更新计划文档与 Git 仓库元数据

## 风险与控制
- 主题覆盖范围扩大后，可能影响部分 Ant Design 默认样式；通过统一变量和构建验证降低风险。
- 导入逻辑改动较大，优先采用兼容型转换，尽量保留旧数据结构。
- 拖拽重排涉及树结构与顺序字段，修改时保持最小闭环，避免顺带改动无关数据模型。
